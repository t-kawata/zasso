# /formulate-tickets

**役割**: 設計書（Requirements / Functional Specification / RFC / 設計ドキュメント）を分析し、依存関係に基づいたフェーズ（段階）・マイルストーン・個別チケットに分解する。各チケットは「1チケット・1不変条件」を徹底し、外部I/Oを排除したメモリ内完結の検証可能な単位に区切る。

## このコマンドの目的

設計書の記述をそのまま実装しようとすると、依存関係の誤った順序や未検証の結合が発生し、後戻りが膨大になる。このコマンドは設計書を以下の観点で分析・分解する：

1. **依存関係の抽出**: 型定義 → トレイト境界 → 純粋関数 → 非同期ランタイム → 統合の5層の依存グラフを構築する
2. **フェーズ分割**: 外部依存（I/O、LLM、DB、乱数）を段階的に導入する順序でフェーズを設計する
3. **チケット詳細化**: 各チケットに対し「不変条件＋実装スコープ＋テスト検証＋計装方法」を精密フォーマットで記述する

## 引数の解釈

- **第1引数（必須）**: 設計書のファイルパス（プロジェクトルートからの相対パス、または絶対パス）
  - 例: `docs/RFC-001-process-registry.md`
  - 例: `/absolute/path/to/design-doc.md`
- **第2引数（省略可）**: 出力先ファイルパス
  - 例: `docs/tickets/RFC-001-tickets.md`
  - 例: `/absolute/path/to/output.md`
  - 省略時は標準出力に出力する

## 出力先

- 第2引数が指定された場合: 指定されたファイルパスにMarkdownを書き出す
- 第2引数が省略された場合: 標準出力にMarkdownを出力する

## 分析手順

### Step 0: 初期化（引数パース）

```bash
_R="$(git rev-parse --show-toplevel)/.claude"

# 第1引数: 設計書パス（必須）
DOC_PATH="${ARGUMENTS%% *}"

# 第2引数: 出力先パス（省略可）
OUTPUT_PATH=""
if [[ "$ARGUMENTS" == *" "* ]]; then
  OUTPUT_PATH="${ARGUMENTS#* }"
fi
```

### Step 1: 設計書の検証と情報抽出

```bash
# ファイル存在確認
if [ ! -f "$DOC_PATH" ]; then
  echo "エラー: 指定された設計書が見つかりません: $DOC_PATH"
  exit 1
fi
```

設計書を読み取り、以下の情報を抽出する：
1. **タイトルと概要**: 設計書の目的、スコープ、技術スタック
2. **型定義**: 構造体、列挙型、トレイト、型エイリアス — すべてのデータ型をリストアップする
3. **関数シグネチャ**: 公開API、非公開関数、メソッド — 引数・戻り値・asyncの有無
4. **依存関係グラフ**: 型Aが型Bに依存 → チケットB→Aの順序で実装
5. **外部依存**: ネットワークI/O、ファイルI/O、LLM、DB、乱数生成 — 後段フェーズに回す
6. **内部依存**: 関数Aが関数Bを呼ぶ、型Xが型Yをフィールドに持つ — 先行実装が必要

### Step 2: CLAUDE.md の自動生成 — 設計全体マップの作成

設計書と同階層に `CLAUDE.md` を生成する。このファイルは個別チケットの作業中に設計全体を
俯瞰するためのマップとして機能する。

```bash
# CLAUDE.md の出力先を決定（設計書と同階層）
DOC_DIR="$(dirname "$DOC_PATH")"
CLAUDE_MD="$DOC_DIR/CLAUDE.md"

# 既存ファイルの上書き確認
if [ -f "$CLAUDE_MD" ]; then
  echo "注意: $CLAUDE_MD は既に存在します。/formulate-tickets により上書きされます。"
fi
```

Step 1 で抽出した情報をもとに、以下の内容で `CLAUDE.md` を生成する：

```bash
cat <<'CLAUDE_EOF' > "$CLAUDE_MD"
# <設計書タイトル> — 設計全体マップ

> このファイルは `/formulate-tickets` によって自動生成されました。
> **生成元:** <設計書パス>
> **生成日:** <現在日付>

## 目的とスコープ

<設計書の目的・スコープの要約>

## アーキテクチャ概要

<主要コンポーネントとその責務の一覧>

## 主要な型とデータ構造

<主要な型・トレイト・構造体とそれらの関係性>

## モジュール／コンポーネント間の関係

<設計書に記述された各コンポーネント・モジュール間の依存関係と結合の一覧>
CLAUDE_EOF
echo "設計全体マップを $CLAUDE_MD に生成しました"
```

生成された `CLAUDE_MD` はチケット作業中の任意のタイミングで Claude Code が自動的に読み込み、
設計全体のコンテキストとして利用される。これにより「木（個別チケット）」の作業中も
「森（設計全体）」を常に念頭に置くことができる。

---

### Step 3: 依存グラフの構築（5層モデル）

抽出した全要素を以下の5層に分類する。**例の部分は設計書から実際に抽出した要素で置き換えること**：

| 層 | 内容 | 外部依存 |
|---|---|---|
| Layer 0（型定義） | 構造体、列挙型、トレイト | なし |
| Layer 1（純粋関数） | 外部I/Oなしの純粋ロジック | なし |
| Layer 2（非同期ランタイム） | async関数、タスク管理 | 非同期ランタイム（tokio等） |
| Layer 3（ライフサイクル管理） | 複数リソースの調整 | 非同期ランタイム |
| Layer 4（統合・プラットフォーム） | 外部システム結合、UI | プラットフォームAPI |

この層構造に従い、Layer 0 → Layer 1 → ... → Layer 4 の順にマイルストーンを設定する。

### Step 4: フェーズ・マイルストーン設計

依存グラフに基づき、段階的に外部依存を導入する順序でフェーズを設計する：

**第1段階: 純粋ロジック・状態機械の完全隔離検証**
- 外部からの揺らぎ（I/O、乱数、非同期実行）を完全に遮断
- Layer 0（型定義）と Layer 1（純粋関数）のみ
- 全テストがメモリ内完結・決定論的・ミリ秒単位

**第2段階: Mock / Fake による制御された実行の導入**
- 非同期ランタイムを導入するが、実I/OはMock/ Fakeで代替
- 仮想クロックで時間を制御
- タイムアウト・競合状態の検証
- Layer 2（非同期ランタイム）が対象

**第3段階: ライフサイクル・エラー処理の統合**
- 複数リソースの協調動作
- 再起動ループ・グレースフルシャットダウン
- 異常系（クラッシュ、タイムアウト）の網羅
- Layer 3（ライフサイクル管理）が対象

**第4段階: プラットフォーム固有・統合・E2E**
- プラットフォーム別実装
- 外部システム結合、UI統合
- クロスプラットフォームE2E検証
- Layer 4（統合・プラットフォーム）が対象

各フェーズ内で、依存関係のある単位をマイルストーンとして区切る。マイルストーンはアルファベットと数字の組み合わせでIDを付与する（例: `M0`, `M1`, ..., `M4`）。

### Step 5: チケットの詳細化

各マイルストーン内のチケットは、依存関係と責務の境界に従って「1チケット・1不変条件」を徹底し、以下のテンプレートに従って記述する：

```markdown
#### チケット <Phase-ID>: <タイトル>

* **参照設計書:** <設計書パス> (§<該当セクション番号>)
* **依存・関連チケットID:** <依存するチケット・依存されるチケットのIDと関係の種類（先行実装必須/後続/並列可能/リソース共有等）>
* **対象不変条件 / 規範:** 設計書の該当セクション番号・仕様箇所。複数ある場合は「、」で列挙。
* **実装の背景と目的:** なぜこのチケットが存在するのか。前段チケットとの関係、将来の差し替え計画、設計判断の根拠を明確に述べる。
* **実装スコープ:**
  - 実装する具体的な型・関数・トレイトの一覧
  - 各要素の責務の簡潔な説明
  - エラー型・バリアントの追加がある場合は明記
  - Fake / Mock 実装を含む場合はその動作仕様
* **テストコードによる検証:**
  1. 正常系の検証項目（正しい入力→正しい出力）
  2. 異常系の検証項目（不正な入力→適切なエラー）
  3. 境界値の検証項目（空、最大、最小、闘値）
  4. 決定論性の検証項目（同一入力→同一出力）
  5. コンパイル時検証（トレイト境界充足、オブジェクト安全性）
* **計装方法・観測対象:** 何を観測し、どのような統計的・代数的性質を確認するか。物理時間ではなく仮想命令ステップ数やエントロピーなど、決定論的な計装指標を用いる。
```

### Step 6: チケットIDの採番

チケットIDは以下の命名規則に従う：

- **マイルストーンID**: `M0`, `M1`, `M2`, ...（フェーズを通して連番）
- **チケットID**: `M<数字>-<連番>`（例: `M0-1`, `M0-2`, ...）
- サブチケット: `M<数字>-<連番>.<サブ番号>`（例: `M0-1.5` — 抽象トレイト階層など、依存関係上は同じ層に属するが別チケットに分割する必要がある場合）

採番は依存順序を反映すること。つまり、先に実装すべきチケットほど小さいマイルストーン番号・チケット番号を持つ。

必要に応じて抽象トレイトを定義するチケットを先行配置する（例: `M-2`, `M-1` のようにマイナス番号で事前準備フェーズを表現してもよい）。

### Step 7: 出力

分析結果を以下の構造でMarkdown出力する。出力先は第2引数の有無で異なる：

**第2引数あり** — 出力先ファイルに書き出す：
```bash
# 生成したMarkdownを OUTPUT_PATH に書き出す
cat <<'TICKETS_EOF' > "$OUTPUT_PATH"
# <設計書タイトル> 実装チケット分解設計書

> **生成元:** <設計書パス>
> **生成日:** <現在日付>
> **分析済みセクション:** <セクション一覧>

---

## <フェーズ名>

> **外部依存:** <このフェーズで使用する外部ライブラリ・API>

### <マイルストーン名>

> **DB:** メモリ内完結（または使用するDB）

#### チケット <ID>: <タイトル>
...
TICKETS_EOF
echo "チケット分解結果を $OUTPUT_PATH に書き出しました"
```

**第2引数なし** — 標準出力に出力する（パイプやリダイレクトで利用可能）：
```bash
# 生成したMarkdownを標準出力に出力する
cat <<'TICKETS_EOF'
# <設計書タイトル> 実装チケット分解設計書
...
TICKETS_EOF
```

---

## 注意事項

- 本コマンドは**設計書の解析結果をチケットとして生成する**に過ぎない。生成されたチケットの内容は、必要に応じてユーザーと調整・修正すること。
- すべての例示は設計書の内容を説明するためのプレースホルダであり、実際の出力では与えられた設計書から抽出した情報で置き換えること。
- 出力ファイルが既に存在する場合は、上書きする前にユーザーに確認を取ること。
