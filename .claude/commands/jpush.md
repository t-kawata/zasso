# JPush — 日本語コミット＆プッシュ

前回のコミット以降の変更を `git diff` / `git status` で分析し、変更内容を日本語で説明するコミットメッセージを生成して `PUSH_MSG="..." make push` を自動実行する。

## Process

### Step 1: 変更状態を確認

```bash
git status --short
git diff --stat
git log --oneline -5
```

`git status --short` が空の場合は「変更はありません」と報告して終了。

### Step 2: 変更内容を詳細に分析

```bash
git diff
git diff --cached
git status
```

出力を読み、以下の観点で分類する：
- 新規追加されたファイル・機能
- 修正された既存コード
- 削除されたファイル
- 設定や依存関係の変更
- テストやドキュメントの変更

### Step 3: 日本語コミットメッセージを生成

分類結果をもとに、以下のフォーマットで日本語のコミットメッセージを生成する：

```
<prefix>: <日本語での変更概要>

- **変更内容**: <修正・追加・削除したものを列挙>
- **変更理由**: <なぜその変更が必要だったか>
- **影響範囲**: <変更が及ぶ箇所や注意点>
- （必要に応じて補足行を追加）
```

**subject（1行目）のルール**:
- 先頭は prefix + `:` + スペース（例: `feat: `）
- 変更内容を日本語で簡潔に（72文字以内）
- 「何ができるようになったか」「何が直ったか」を一言で表現

**body（2行目以降）のルール**:
- subject と body の間に空行を1行入れる
- body は Markdown の箇条書き（`- `）で記述
- **変更内容**: 具体的に修正・追加したファイルや処理を列挙
- **変更理由**: なぜこの変更が必要だったか、どのような問題・要件があったかを説明
- **影響範囲**: この変更によって影響を受ける可能性のある機能や注意点

**prefix の種類**（conventional commits に準拠）:

| prefix | 使用場面 |
|--------|---------|
| `feat` | 新機能・新規ファイル追加 |
| `fix` | バグ修正 |
| `refactor` | 動作変更を伴わないコード整理 |
| `docs` | ドキュメントのみの変更 |
| `test` | テストの追加・修正 |
| `chore` | ビルド・設定・依存関係 |
| `perf` | パフォーマンス改善 |
| `style` | フォーマット・スタイルのみの変更 |

例:
```
feat: ログアウトボタンを追加

- **変更内容**: ヘッダー領域にログアウトボタンを追加。クリック時に確認ダイアログを表示し、承認後セッションを破棄してログイン画面へ遷移する
- **変更理由**: ユーザーからアカウント切り替えの要望が複数寄せられていた。これまではブラウザのCookie削除が必要だった
- **影響範囲**: 認証状態を保持するSessionStoreの破棄処理が新たに必要。ログイン画面へのリダイレクトパスに注意

fix: ユーザー名に特殊文字を含む場合の認証エラーを修正

- **変更内容**: ユーザー名のバリデーション時に @ や # を含む文字列で認証が失敗するバグを修正。SQLクエリのエスケープ処理を見直した
- **変更理由**: 特殊文字を含むユーザー名（例: "user@company"）でログインできないという報告が本番運用で発生
- **影響範囲**: 認証モジュールのsanitize処理のみ。既存の正常なユーザー名には影響しない
```

### Step 4: リリースノートを生成

Step 2 の分析結果をもとに、MYCUTE プロジェクトのリリースノートを英語で生成する。

```bash
# MYCUTE プロジェクトの次のバージョン番号を取得
cd ~/shyme/mycute && make next-version
```

手順:
1. `cd ~/shyme/mycute && make next-version` を実行し、次のバージョン番号を取得する（例: `v0.24.42`）
2. `~/shyme/mycute/release-notes/` ディレクトリが存在しない場合は作成する
3. `~/shyme/mycute/release-notes/<バージョン>.md` にリリースノートを書き出す（例: `v0.24.42.md`）

リリースノートのフォーマット:

```markdown
# Release v<version>

## Summary
<英語で変更の概要を1〜2文>

## What's Changed
<GitHub Releases に記載する形式で変更点を英語で箇条書き>
- <新機能>
- <バグ修正>
- <リファクタリング／改善>
```

**内容生成のルール**:
- 言語は **英語**（GitHub の国際的な公開リリースノートとして使用されるため）
- 概要（Summary）は 1〜2 文で、今回のリリースの要点を伝える
- What's Changed は Step 2 の分析結果を元に、変更の種類ごとに整理して箇条書き
- ファイル名は `v` から始め、バージョン番号 + `.md`（例: `v0.24.41.md`）
- 書き出す前に既に同名ファイルが存在する場合は上書きしてよい

### Step 5: リリースノートを MYCUTE リポジトリにコミット＆プッシュ

生成したリリースノートを MYCUTE プロジェクトのリポジトリにコミットし、プッシュする。

```bash
cd ~/shyme/mycute
git add release-notes/
git commit -m "docs: add release notes for <バージョン>"
git push origin master
```

### Step 6: 変更をコミット＆プッシュ

```bash
PUSH_MSG="<Step 3 で生成したコミットメッセージ>" make push
```

### Step 7: 結果報告

実行結果を報告する。失敗した場合はエラー内容を表示して終了。

## Edge Cases

- **`make push` が定義されていない**: Makefile が存在しない、または `push` ターゲットがない場合。「Makefile に push ターゲットが見つかりません」と報告して終了。
- **変更量が膨大**: diff の全出力が巨大な場合、`git diff --stat` を優先し、詳細は主要ファイルのみ確認する。
- **コンフリクト残留**: `git status` に `UU` や `AA` などのコンフリクトマーカーがある場合、「コンフリクトが解消されていません」と報告して終了。
