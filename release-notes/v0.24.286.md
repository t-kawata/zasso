# Release v0.24.286

## Summary

Completed Phase 7 (M8) — full hotkey-driven speech input pipeline crate-internal. Added hotkey monitoring (macOS CGEventTap / Windows rdev+WH_KEYBOARD_LL), clipboard+keyboard injection, Voiput integration with enable_hotkeys/build_flush_text/request_flush, test-run restructured with CLI engine selection, and fixed native callback dispatch (removed MainActor.run which was blocked by Tokio runtime).

## What's Changed

- **feat**: Add `hotkey/` module (M8-1) — Option/Alt double-tap detection, Ctrl+Option orchestator combo via CGEventTap (macOS) and rdev+GetAsyncKeyState+WH_KEYBOARD_LL (Windows)
- **feat**: Add `input/` module (M8-2) — arboard clipboard wrapper, CGEvent (macOS) / SendInput (Windows) keyboard injection with save_paste_and_restore, input_diff
- **feat**: Voiput extension (M8-3) — enable_hotkeys(), build_flush_text(), request_flush(), paste_at_cursor(), InputMode (Buffered/RealTime), flush_tx 4-stage dispatch
- **feat**: test-run.rs restructured (M8-4) — 3-stage pipeline (tests → Voiput → event loop), CLI args (--engine, --locale, --openai-key, --base-url)
- **fix**: RECORDING_ACTIVE flag sync (M8-3fix) — process_hotkey_action() now calls mac::/win::set_recording_active()
- **fix**: Add permission guide in request_permissions() — log::warn! display + OS settings launcher (open/start ms-settings:)
- **fix**: SpeechHelper.swift — removed await MainActor.run from Tahoe result callback (blocked by Tokio runtime), rebuilt libSpeechHelper.a
